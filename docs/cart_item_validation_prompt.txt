The cart service is currently using event processing to validate products as they are added to a cart by the user of a front-end application.  This was implemented as the primary validation process because 
the architecture does not allow separate domain microservices to read the daatabases of other services nor does it allow the services to use http services to communicate with each
other.  The architecture uses the SAGA pattern.  

We recently implemented the plan detailed in @docs/order-secure-api-plan-v4.md. We ran into a simialr situation in which the order process needed to be able to validate customer data
for security purposes without directly reading the customer database or using http services internally.  To resolve that we implemented a cache of the necessary customer data that is
initialized on startup of the order service and is maintained through the handling of customer_created and customer_updated events.  That is the primary mechanism used to check customer
information by the order service.  We do use events as a secondary mechanism if the cache fails.

I would like to do something similar for the cart service to make product validation a little quicker.  Learn how the order and customer services have implemented the cache and secondary event
mechanism, evaluate the current product validation flow used by the cart and product services, and then produce a plan to implement that process for the cart and product services for product 
validation.  There may be other services in which having a product cache could be useful, so keep that in mind as you develop the plan.
